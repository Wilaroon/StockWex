Objetivo de la Fase 1

Construir el núcleo del sistema StockWex, enfocado en la gestión de productos, inventario y pedidos, estableciendo una base sólida, escalable y bien estructurada para futuras expansiones.

Esta fase representa el corazón del sistema, donde se define la lógica principal del negocio.


Filosofía de esta fase

Esta fase define la base del sistema, no todas las funcionalidades finales.
La estructura debe priorizar escalabilidad y mantenibilidad.
Las decisiones importantes deben quedar documentadas en docs/decisions/ (ADR).
El roadmap es una guía estratégica, no un contrato fijo. Puede ajustarse según necesidades reales del proyecto.

Módulos principales


1. Gestión de Productos (Core de catálogo)
Objetivo

Administrar toda la información base de los productos del sistema.

Funcionalidades
Registro de productos.
Edición de productos.
Eliminación lógica de productos.
Gestión de marcas.
Gestión de departamentos.
Gestión de tipos de productos.
Gestión de unidades de medida.
Asociación de múltiples imágenes por producto.
Identificador interno (id) y público (uuid).
2. Gestión de Inventario
Objetivo

Controlar el stock real de los productos en el sistema.

Funcionalidades
Control de stock por producto.
Stock mínimo para alertas.
Visualización de inventario general.
Validación de stock en operaciones.
3. Gestión de Pedidos
Objetivo

Permitir la creación y administración de pedidos internos.

Funcionalidades
Creación de pedidos.
Cálculo automático de totales.
Registro de observaciones.
Gestión de estados de pedidos.
Actualización automática del inventario al crear o cerrar pedidos.
4. Infraestructura del Sistema
Objetivo

Establecer la base técnica del backend.

Funcionalidades
API REST con FastAPI.
Arquitectura modular del backend.
Conexión a PostgreSQL.
Migraciones con Alembic.
Uso de SQLAlchemy como ORM.
Validaciones con Pydantic.
Contenerización con Docker.
Reglas del sistema en esta fase
El inventario no depende del estado del producto, sino de su stock.
Todo producto debe tener trazabilidad mediante created_at y updated_at.
Los cambios en base de datos deben ser gestionados con migraciones (Alembic).
Las imágenes no se almacenan en la base de datos, solo sus rutas.
Las decisiones arquitectónicas importantes deben documentarse en ADR.
